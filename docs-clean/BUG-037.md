## Summary

`CKnownFileList::SafeAddKFile()` in current `main` still resolves a same-hash collision by
destructively replacing the old entry with the new one.

That is not just theoretical debt. The current source already carries an in-code warning
that this "may lead to unshared files" and may leave files "shared" in the GUI while not
shared logically.

## Current Mainline Evidence

The live same-hash path in `srchybrid/KnownFileList.cpp` still does all of this:

- keys the collision on MD4 hash alone
- removes the old known-file entry
- may remove the old shared-file entry
- merges some statistics
- deletes the old object and keeps the new one

That is fragile when two on-disk files legitimately share the same hash or when the same
file is rediscovered under a new path.

## Why This Matters

Representative low-drama failure cases:

- a shared file is moved between shared directories
- the same hashed file exists in two shared directories
- startup or reload rediscovers an equivalent file before share state fully settles
- a previously downloaded/shared file is reintroduced through another path

Those are exactly the kinds of cases that can create unshared files, mismatched GUI
state, or accidental loss of the authoritative shared instance.

## Comparison Notes

- `analysis\emuleai` goes much further with duplicate-path/history tracking and shows
  that the problem space is real
- the focused Xtreme mod archive still carries the historical warning on this surface,
  which suggests the logical flaw has been known for a long time

The branch does not need eMuleAI's full duplicate-history feature set to justify fixing
the core destructive replacement behavior.

## Future Option

If the narrow MD4-only fix still leaves too much ambiguity, a local strong-hash sidecar
is a viable later stabilization option:

- keep MD4 as the protocol and `known.met` identity
- add a separate local cache keyed by path / size / mtime with a strong content hash such
  as BLAKE3
- use that sidecar only to distinguish true same-content rediscovery from local same-MD4
  ambiguity before merging or replacing `KnownFile` state

That should be treated as a local persistence/consolidation aid, not as a first-pass
protocol change.

## Intended Fix Direction

Keep the fix narrow and stock-friendly:

1. stop replacing entries solely because the MD4 matches
2. preserve the authoritative live shared/on-disk object when path/share state would
   otherwise be lost
3. merge statistics without deleting the only logically shared instance
4. retain enough alternate-path knowledge to avoid GUI/logical share mismatches
5. do not import the full duplicate-history UI unless later evidence says it is needed
6. do not require a new sidecar hash format for the first fix; treat any BLAKE3-style
   sidecar as a later optional hardening layer

## Validation

- move a shared file between two shared directories and verify it stays shared
- keep duplicate same-hash files in two shared directories and verify neither is silently
  unshared
- restart after duplicate discovery and verify share state and uploads remain correct
- verify known-file statistics survive the non-destructive merge path
