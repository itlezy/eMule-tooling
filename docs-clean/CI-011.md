## Summary

Make the broadband release live E2E lane a first-class release gate instead of
an ad hoc collection of scripts.

## Release 1.0 Classification

**Release Gate.** This owns the operator-facing first-release command and the
result artifact shape. It should aggregate the release-critical REST,
WebServer, live-wire, aMuTorrent, and Arr controller checks without making every
future UI E2E lane block `emule-bb-v1.0.0`.

The aggregate runner should cover:

- UI flows for preferences, shared files, search, and download lifecycle
- REST contract and live-wire checks
- WebServer robustness checks
- aMuTorrent integration coverage where a live eMule BB instance is required
- Prowlarr, Radarr, and Sonarr integration coverage through the live Arr suite

## Acceptance Criteria

- [ ] `run_live_e2e_suite.py` has stable selectable suites for every release
      lane
- [ ] each suite writes a result artifact with status, command line, timings,
      and failure phase
- [ ] public network failures are reported as inconclusive with diagnostics
      when the app and harness behaved correctly
- [ ] the default release lane uses isolated temp profiles and does not depend
      on user profile state
- [ ] aMuTorrent, Prowlarr, Radarr, and Sonarr suites are included in the
      default release lane and produce redacted artifacts
- [ ] release documentation points operators to one supported command
