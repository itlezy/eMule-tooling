## Summary

Community-0.72 (irwir) upgraded from MbedTLS 3.x to **MbedTLS 4.0** in commit `24d1de7`
(January 2026). MbedTLS 4.0 is a breaking API release with several structural changes.

Our codebase is currently on a pinned MbedTLS 3.x submodule. The upgrade is needed to:
- Stay current with security patches (MbedTLS 3.x support timeline)
- Pick up TLS 1.3 support (community-0.72 added this in commit `1398352`)
- Align with the ARM64 build path (community's ARM64 platform requires MbedTLS 4.0)
- Remove deprecated API usage that 4.0 drops

## Breaking API Changes (from community diff)

Community's `mbedtls/include/mbedtls/threading_alt.h` shows the key structural change:

```cpp
// 3.x:
typedef struct {
    CRITICAL_SECTION cs;
    char is_valid;
} mbedtls_threading_mutex_t;

// 4.0:
typedef struct {
    CRITICAL_SECTION cs;
    char is_valid;
} mbedtls_platform_mutex_t;           // renamed

typedef struct {
    int cond;
} mbedtls_platform_condition_variable_t;  // NEW
```

The file was also relocated from `mbedtls/include/mbedtls/` to
`mbedtls/tf-psa-crypto/include/mbedtls/` — MbedTLS 4.0 reorganizes its PSA crypto
subsystem into a separate directory structure (`tf-psa-crypto/`).

Additional changes expected:
- `MBEDTLS_ALLOW_PRIVATE_ACCESS` usage (our codebase uses this — may be affected)
- TLS context initialization APIs
- Entropy/RNG API changes

## TLS 1.3 (related)

Community-0.72 added TLS 1.3 support in commit `1398352`. This is enabled by MbedTLS 4.0's
TLS 1.3 support. Our `EncryptedStreamSocket.cpp` and MbedTLS integration would need the
corresponding TLS 1.3 enable flags after the upgrade.

TLS 1.3 for eMule is primarily relevant for:
- Web interface HTTPS (if re-introduced)
- Future: potential TLS upgrade of the server connection (not standard eDonkey protocol)
- Reduced TLS handshake latency for any TLS-wrapped connections
