## Summary

MKV and WebM share the same EBML magic header `1A 45 DF A3`. The current
`GetMimeType()` magic-byte table returns `video/x-matroska` for both, which
is technically incorrect for WebM files.

This is an optional quality-of-life improvement — returning `video/x-matroska`
for WebM files is not harmful for the file-info dialog use case, but it
produces incorrect MIME types when the information is used elsewhere.

## Fix

Parse the EBML DocType element deeper in the header (typically within the first
64–256 bytes) to distinguish:
- `webm` → return `video/webm`
- `matroska` → return `video/x-matroska`

## Acceptance Criteria

- `.webm` files return `video/webm` from `GetMimeType()`.
- `.mkv` files return `video/x-matroska`.
- No regression on existing format types.

## Files

- `srchybrid/OtherFunctions.cpp` — `GetMimeType()` implementation (EBML entry in
  the magic-byte table added in commit `2f026c7`)

## Experimental Reference Implementation

**Status in `stale-v0.72a-experimental-clean`:** Done. The fix is in `srchybrid/MediaInfo.cpp` (not `OtherFunctions.cpp` — the experimental branch moved MIME detection to `MediaInfo.cpp` as part of the MediaInfo unification work). The EBML DocType is parsed:
```cpp
static const char _cDocTypeWebM[] = "webm";
static const char _cDocTypeMatroska[] = "matroska";
// returns "video/webm" or "video/x-matroska" based on DocType element content
```
The magic byte table entry uses `SMimeMagicEntry::MMT_Ebml` type with a special handler that reads the EBML DocType string from within the first ~256 bytes.

**Porting note:** If MediaInfo.cpp from experimental is not being ported, the equivalent fix can be added to `OtherFunctions.cpp::GetMimeType()` directly by adding a post-match EBML DocType probe in the `video/x-matroska` branch.
