## Summary

Create a machine-readable REST contract manifest and use it as the canonical
test input for the live REST completeness lane.

The current Python route tuple is useful, but it is not a durable public
contract artifact and can drift from `docs/REST-API-CONTRACT.md`.

## Acceptance Criteria

- [ ] a checked-in contract manifest lists every public `/api/v1` route,
      method, safety classification, and expected response envelope
- [ ] route seam tests and live REST smoke consume or validate against the same
      manifest
- [ ] `docs/REST-API-CONTRACT.md` is checked against the manifest for route
      drift
- [ ] adding a new REST endpoint requires updating the manifest and tests
- [ ] `POST /api/v1/app/shutdown` remains explicitly excluded from live
      completeness mutation loops

## Progress

- 2026-05-02: Added the first checked-in machine-readable target contract at
  `docs/REST-API-OPENAPI.yaml` and a legacy runtime-action parity inventory at
  `docs/REST-API-PARITY-INVENTORY.md`. The CI item remains open because route
  seam tests and live REST smoke do not yet consume the contract.
- 2026-05-02: Expanded native route seam coverage and live REST smoke route
  coverage for server/Kad operations, transfer details, shared-file metadata,
  and search result download/delete-all routes. The CI item remains open until
  the smoke runner consumes the checked-in OpenAPI contract directly.
